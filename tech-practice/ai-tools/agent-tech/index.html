    Agent 技术串讲¶

    从 LLM 到 Agent 的完整技术脉络。

    本文整理自技术分享文档 《近年 AI 应用技术串讲与优质文档分享》。

    1. LLM：大语言模型¶

    核心论文：Attention Is All You Need（Vaswani et al., 2017）

    Transformer 架构的提出奠定了大模型时代的基础，使基于注意力机制的生成模型成为主流。

    三种架构变体：
┌─────────────────┐  ┌──────────────────────┐  ┌─────────────────┐
│  Encoder-Only   │  │  Encoder-Decoder     │  │  Decoder-Only   │
│  (BERT 等)       │  │  (T5, BART 等)       │  │  (GPT, LLaMA 等) │
│  🟣 理解任务     │  │  🟢 经典生成架构      │  │  🔵 当前最主流   │
└─────────────────┘  └──────────────────────┘  └─────────────────┘

    关键词：Self-Attention、Multi-Head Attention、Positional Encoding、Transformer Block

    2. Prompt Engineering：提示词工程¶

    参考：提示词工程笔记

    提示词（Prompt）是用来引导模型按照特定意图生成输出的输入指令。主要包含两类：

    类型 作用 示例
    系统提示词 设定角色、规则、输出格式 "你是一个严谨的代码审查员"
    用户提示词 提出具体问题或任务 "审查这段代码的安全性"

    核心思想：通过设计和优化提示词，使大模型更准确、可控地产生所需输出。是提升效果但不改变模型参数的低成本调优手段。

    常用技巧：

    技巧 说明
    Few-shot 给几个示例让模型学会格式
    Chain-of-Thought 让模型「一步步思考」
    Role Prompting 赋予角色身份
    Structured Output 明确要求 JSON/Markdown 格式

    3. Fine-tuning：微调¶

    核心论文：LoRA: Low-Rank Adaptation of Large Language Models（Hu et al., 2021）

    微调是在已有模型基础上，用特定数据再训练，让模型更适合某个具体任务或场景。

    LoRA 的核心思想¶

    全参数微调：训练所有 W（昂贵）
LoRA 微调：  W' = W + ΔW = W + A·B
             只训练 A 和 B（低秩矩阵），大幅降低训练成本

    优势：

    • 参数量减少 1000 倍以上
    • 可在消费级 GPU 上微调 7B/13B 模型
    • 多个 LoRA 适配器可热切换

    常见框架：Hugging Face PEFT、Unsloth、Axolotl

    4. RAG：检索增强生成¶

    核心论文：Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks（Lewis et al., 2020）

    先从外部知识库检索相关信息，再结合这些信息一起生成回答，从而提升模型的准确性和知识时效性。

    用户提问 → 检索相关文档 → [文档 + 问题] → LLM → 回答

    典型架构：

    组件 常用方案
    向量数据库 Pinecone, Chroma, Milvus, Weaviate
    Embedding 模型 text-embedding-3-small, bge-large
    检索策略 语义检索 + 关键词混合（Hybrid Search）
    生成模型 GPT-4, Claude, Qwen

    适用场景：知识问答、文档分析、客服系统

    5. Function Calling：函数调用¶

    Function Calling 是让大模型按约定格式输出调用指令，从而由外部系统真正去执行具体操作的一种机制。

    用户："今天北京天气怎么样？"
模型：不直接回答 → 输出 {"function": "get_weather", "args": {"city": "北京"}}
系统：执行 get_weather("北京") → 返回结果
模型：结合结果生成最终回答

    本质：让模型从「只会说话」变为「会调用工具」。

    6. MCP：Model Context Protocol¶

    官方文档：MCP Introduction

    MCP 是一种标准化协议，用来让大模型以统一的方式连接外部工具、数据源和服务。

    传统方式：每个工具单独写胶水代码
MCP 方式： 工具 → MCP Server → 统一的 MCP Client → 任何 AI 应用

    最重要的贡献：使工具可以跨 AI 应用复用，推动社区生态发展。

    典型 MCP Server 示例：文件系统、数据库、浏览器、GitHub、飞书文档……

    7. Agent：智能体¶

    核心论文：ReAct: Synergizing Reasoning and Acting in Language Models（Yao et al., 2022）

    Agent 是一种能够基于目标进行「思考 → 行动 → 观察」循环、能够自主调用工具来完成复杂任务的智能系统。

         ┌──────────┐
     │  思考    │ ← 分析当前状态，决定下一步
     │ (Think)  │
     └────┬─────┘
          ↓
     ┌──────────┐
     │  行动    │ ← 调用工具、执行操作
     │  (Act)   │
     └────┬─────┘
          ↓
     ┌──────────┐
     │  观察    │ ← 获取工具返回结果
     │(Observe) │
     └────┬─────┘
          ↓
     回到「思考」

    最小 Agent 公式：

    提示词 + LLM + Tools = Agent

    Agent 设计模式（参考 AI Agent Workflow Design Patterns）：

    模式 说明
    ReAct 交替推理和行动
    Plan & Execute 先规划再执行
    Reflection 执行后反思改进
    Tool Use 调用外部工具
    Routing 根据意图分发到不同处理器

    8. Multi-Agent：多智能体¶

    参考：Building Multi-Agent Systems（Anthropic）

    由多个分工协作的 Agent 共同完成任务，通过拆分任务与隔离上下文解决单 Agent 系统难以处理的复杂问题。

    主 Agent（协调者）
    ├── Agent A：负责 XX 子任务
    ├── Agent B：负责 YY 子任务
    └── Agent C：负责 ZZ 子任务
         ↓
    汇总结果 → 最终输出

    优势：上下文隔离、专业化分工、并行处理

    需要警惕：

    • Token 消耗量大
    • 协作效率可能低
    • 系统复杂度高
    • 不要为了多 Agent 而多 Agent——先确认单 Agent 确实搞不定

    9. Context Engineering：上下文工程¶

    参考： - LangChain Context Engineering - Effective Context Engineering for AI Agents（Anthropic）

    Agent 运行中需要提供给 LLM 的一切相关信息都是上下文：对话历史、用户输入、背景知识、工具结果、系统提示词……

    核心关注点：

    操作 目标
    筛选 只保留对当前决策有用的信息
    压缩 减少 Token 消耗，提高效率
    组织 按优先级排列，关键信息靠前

    类比：上下文工程 ≈ 给 LLM 做「信息整理」——不是给越多越好，而是给最关键的。

    总结：技术栈层级¶

    Layer 5   Multi-Agent    ← 多 Agent 协作
Layer 4   Agent          ← 思考-行动-观察循环
Layer 3   MCP / Tools    ← 标准化工具连接
Layer 2   RAG / Fine-tune ← 知识增强 / 能力定制
Layer 1   LLM + Prompt   ← 基础模型层

    从下往上，每一层建立在前一层之上。理解这个层级是理解整个 AI Agent 生态的关键。
